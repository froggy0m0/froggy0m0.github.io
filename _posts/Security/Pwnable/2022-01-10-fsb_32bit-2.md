## FSB(Format String Bug)-2

### example4

```c
// Name : example4
// Compaile : gcc -o example4 example4.c -m32 -mpreferred-stack-boundary=2 -fno-stack-protector -z execstack -no-pie

#include<stdio.h>


void get_shell()
{
    system("/bin/sh");
}

int main(void)
{   
    int val = 0xfff; // 0xfff = 4095
    char buf[20] ={0,};

    printf("val = %d\n", val);
    printf("val_addr = %p\n", &val);


    gets(buf);
    printf(buf);
    
    if(val == 0xffe) // 0xffe = 4094
    {
        get_shell();
    }
    printf("\nval = %d\n", val);

}
```



![img](https://Froggy0m0.github.io/assets/img/Pwnable/2022-01-10-fsb_32bit-2-1.png)Fig 6. example4 메모리보호기법



**example4** 는 val 변수의 주소를 출력하고 사용자로부터 입력을 받고 출력해주고 val 변수가 'B' 이면 get_shell 함수를 호출한다.

쉘을 얻기 위한 시나리오로는 val변수의값을 0xffe 로 변경 시켜 쉘을 얻거나

PIE는 적용돼있지 않고 Partial RELRO가 적용돼있기 때문에 get_shell함수의 주소를 얻어 print의 GOT 에 overwrite하여 printf함수 대신 get_shell함수를 실행시킬 수 있을 것이다.

**Fig6** 를 통해 **val** 변수 주소(0xffffd3c0)가 고정돼있는 것을 확인해 볼 수 있다.

먼저 `AAAA %p %p %p %p`를 입력하여 esp와의 offset을 확인해보았다.

실행 결과 AAAA(0x41414141)은 `%p`로 `%1$p`에 위치해있다.

#### val 변수값 변경

먼저 val 변수값을 변경하여 쉘을 얻어보겠다.

 AAAA의 위치에 val변수의 주소를 넣고 %[숫자]c를 사용하여 '0xffe'의 ASCII값 0x42 만큼 채워놓은후 $[offset]$n으로 val 변수를 'B'를 채워놓을수 있을 것이다.

이해를 돕기 위해 해당 __example3__ 를 gdb로 실행시켰다.

main+29 부터  main+57까지 a배열에 해당하는 값들이 역순으로 스택에 push되었다. 

순서대로 push 된 것처럼 보이지만 __mov    dword ptr [ebp - 0x1c], 0x6c__ 로 역순으로 push되고 있다. (스택에서도 역순으로 push된게 보이지만 설명하자면) 인덱스가 낮을수록(index 0 = 'l') ebp에서 멀어지고(esp와는 가깝고) 인덱스가 높아질수록(index 4 = 'e')ebp와 가까워진 것을 볼 수 있다.배열 마지막 인덱스 에 해당하는 101은 아직 저장되지 않았지만 0xffffd35c 에 0x65로 저장되어진다.

GDB를 계속 실행시켜 마지막 printf인 `main+86` 까지 진행 시켰다.

main+86의 esp에서 현재 esp(0ffffd348) 로부터 계산되어진 결과 __%1$n__ = `l`, __%2$n__ = `h`, __%3$n__ = `o`, __%4$n__ = `l`, __%5$n__ = `e`로 포맷 되어 h e l l o 라는 출력 값을 받아 볼 수 있게 된다.
